## 1. 项目背景与目标

### 1.1 背景

抖音平台日活用户超7亿，大量潜在客户聚集在热门视频的评论区。传统人工监控评论、手动私信的方式效率极低，无法实现规模化获客。本产品旨在通过自动化手段，帮助营销人员从竞品或行业视频评论区中批量获取意向用户，并完成私信转化。

### 1.2 产品目标

| 目标类型 | 描述 |
|----------|------|
| 核心目标 | 实现评论监控 → 用户识别 → 私信转化的全链路自动化 |
| 效率目标 | 相比人工操作，获客效率提升 10x 以上 |
| 稳定目标 | 支持 7×24 小时无人值守运行 |
| 安全目标 | 多重机制防止账号被限流或封禁 |

### 1.3 核心价值主张

```
监控竞品评论区  →  识别意向用户  →  自动私信转化  →  沉淀私域流量
```

## 2. 用户画像

### 2.1 主要用户群体

**用户 A：独立营销人员**
- 角色：个人卖家、代运营从业者
- 痛点：人力有限，无法持续监控多个视频评论区
- 核心需求：低成本、自动化、稳定运行

**用户 B：MCN 机构 / 营销团队**
- 角色：团队负责人或运营主管
- 痛点：需管理多个账号，多个关键词方向同时运营
- 核心需求：多账号并发、数据可视化、操作可审计

**用户 C：代理商 / 贴牌需求方**
- 角色：希望二次销售或品牌定制的合作伙伴
- 核心需求：可贴牌、可定制功能、有源码或完整授权

### 2.2 使用场景

| 场景 | 操作流程 |
|------|---------|
| 评论截流 | 监控竞品视频 → 过滤意向评论 → 自动私信 |
| 关键词获客 | 搜索行业关键词视频 → 采集评论用户 → 自动私信 |
| 私域引流 | 监控热门视频 → 识别目标用户 → 引导加微信 |
| 品牌维护 | 监控品牌相关视频 → 及时发现负面评论 → 人工介入 |

## 4. 模块一：评论监控系统

### 4.1 视频监控管理

#### 4.1.1 功能描述
用户可手动输入或批量导入抖音视频链接，系统解析视频 ID 并加入监控列表。

#### 4.1.2 功能需求

| 需求编号 | 需求描述 | 优先级 |
|---------|---------|--------|
| CM-001 | 支持输入抖音视频链接，系统自动解析视频 ID | P0 |
| CM-002 | 支持从文本文件批量导入视频链接 | P1 |
| CM-003 | 监控列表展示：视频标题、视频 ID、添加时间、监控状态、累计采集评论数 | P0 |
| CM-004 | 支持对单个视频启动/暂停/删除监控 | P0 |
| CM-005 | 支持一键全部启动/全部暂停 | P1 |
| CM-006 | 监控视频数量无上限 | P0 |
| CM-007 | 关键词搜索到的视频可直接加入监控列表，视频元数据同步保存 | P0 |

#### 4.1.3 视频 ID 解析规则
- 标准链接：`https://www.douyin.com/video/{video_id}`
- 分享短链：`https://v.douyin.com/xxxxx/`（需先重定向解析）
- 纯 ID 输入：直接使用

### 4.2 实时评论抓取引擎

#### 4.2.1 功能描述
系统对监控视频列表中的视频进行轮询，实时抓取新增评论。

#### 4.2.2 功能需求

| 需求编号 | 需求描述 | 优先级 |
|---------|---------|--------|
| CM-010 | 多视频并发轮询，轮询间隔可配置（建议 30~120 秒） | P0 |
| CM-011 | **1级评论**采集字段：评论ID(CID)、抖音号(sec_uid)、用户昵称、评论内容、发布时间、点赞数、IP归属地/地区 | P0 |
| CM-012 | **2级评论（回复）**采集字段：同1级，额外记录父评论CID、被回复人昵称 | P0 |
| CM-013 | 最多采集到2级，不采集3级及更深层回复 | P0 |
| CM-014 | 展示实时监控状态：当前轮次、本轮新增评论数、总评论数 | P0 |
| CM-015 | API 响应失败时自动重试，最大重试次数可配置 | P1 |
| CM-016 | **实时评论流切换控件**：右侧面板提供"显示模式"切换按钮，两种模式：① **仅新增**（默认）：只展示本次运行后新抓取的评论，实时追加到列表顶部；② **全部**：展示所有历史已抓取的评论（含已处理/已发私信的），可用于复查。切换时附有文字说明两者区别，全部模式下已发过私信的评论行用浅灰色标注 | P1 |

#### 4.2.3 评论采集字段规范

| `ip_label` | `region` | IP归属地（如"广东"） |

### 4.3 多维度评论筛选

#### 4.3.1 功能描述
对抓取的评论进行多维度过滤，仅保留符合条件的潜在客户评论。

#### 4.3.2 筛选条件需求

| 筛选维度 | 需求描述 | 优先级 |
|---------|---------|--------|
| 关键词白名单 | 评论内容必须包含指定关键词（支持多个，OR 关系） | P0 |
| 关键词黑名单 | 评论内容不能包含指定关键词（支持多个） | P1 |
| 昵称过滤 | 昵称包含指定词的用户评论不进入队列 | P1 |
| 地区白名单 | 仅保留指定省份/城市的用户评论 | P1 |
| 地区黑名单 | 排除指定省份/城市的用户评论 | P1 |
| 拉黑用户 | 被拉黑的 UID 评论自动跳过 | P0 |
| 自动拉黑 | 同一用户评论出现 N 次后自动加入黑名单（N 可配置） | P1 |
| **拉黑关键词** | 评论内容包含指定词时直接拉黑该用户（区别于过滤：拉黑是永久，过滤仅跳过本条） | P1 |

#### 4.3.3 黑名单管理（CM-026）

| 需求编号 | 需求描述 | 优先级 |
|---------|---------|--------|
| CM-026 | 标题栏显示当前黑名单用户数，数字以红色圆形角标展示 | P0 |
| CM-027 | 点击"黑名单"入口，弹出黑名单管理弹窗 | P1 |
| CM-028 | 黑名单弹窗展示：UID、昵称、加入时间、加入原因（自动/手动/关键词） | P1 |

### 5.3 自动采集

#### 5.3.1 功能描述
系统根据配置，定时对所有启用的关键词执行采集任务，并将符合过滤条件的视频自动添加到监控列表。

#### 5.3.2 功能需求

| 需求编号 | 需求描述 | 优先级 |
|---------|---------|--------|
| VC-020 | 采集间隔可配置（分钟为单位，1-60分钟，默认1分钟） | P0 |
| VC-021 | 最大并发数可配置（1-5 个关键词同时采集，默认3） | P1 |

## 8. 非功能性需求

### 8.1 性能需求

| 指标 | 要求 |
|------|------|
| 评论监控响应 | 评论发布后，系统在下一个轮询周期内捕获（通常 30~120 秒） |

| 私信发送吞吐 | 10 线程并发时，每分钟可处理 20+ 发送任务 |
| 内存占用 | 正常运行时 RAM 占用 < 500MB |
| 启动时间 | 应用冷启动时间 < 5 秒 |

### 8.2 稳定性需求

| 指标 | 要求 |
|------|------|
| 连续运行时间 | 支持 7×24 小时无人值守运行 |
| 崩溃恢复 | 意外崩溃后重启，自动恢复监控任务和发送队列 |
| API 异常处理 | 抖音 API 返回错误时，记录日志并按重试策略处理，不中断其他任务 |

### 8.3 安全需求

| 需求 | 说明 |
|------|------|
| 本地加密存储 | Token、私钥等敏感凭证使用 AES 加密后存储到本地 JSON 文件 |
| 不上传数据 | 所有数据（账号信息、评论数据、私信记录）仅保存在本地，不上传到任何服务器 |
| 证书复用 | 证书缓存 24 小时，减少提取次数降低暴露风险 |

### 8.4 可用性需求

| 需求 | 说明 |
|------|------|
| 界面语言 | 中文简体 |
| 操作反馈 | 所有关键操作均有即时状态反馈（成功/失败/进行中） |
| 帮助文档 | 内置操作说明和常见问题解答 |
| 配置持久化 | 所有配置在关闭应用后保存，重启后自动恢复 |

## 9. 技术架构设计

### 9.1 总体架构

```
┌─────────────────────────────────────────────────┐
│                   前端界面层                      │
│         Wails + Vue3/React (Windows 原生窗口)     │
└─────────────────┬───────────────────────────────┘
                  │ IPC 通信
┌─────────────────▼───────────────────────────────┐
│                   业务逻辑层 (Go)                 │
│  评论监控引擎 | 视频采集引擎 | 私信发送引擎        │
│  关键词管理器 | 过滤引擎    | 任务队列管理器       │
└────────────┬───────────────────┬────────────────┘
             │                   │
┌────────────▼──────┐  ┌─────────▼──────────────┐
│   数据持久化层     │  │    外部接口层           │
│  JSON 本地文件存储 │  │  抖音 API | 比特浏览器 API│
│  AES 加密模块      │  │  钉钉 Webhook           │
└───────────────────┘  └────────────────────────┘
```

### 9.2 核心模块说明

| 模块名称 | 技术说明 |
|---------|---------|
| 评论监控引擎 | Go goroutine 并发轮询，每个视频独立 goroutine，channel 传递结果 |
| 视频采集引擎 | 带优先级队列的并发采集，worker pool 模式（最大 10 workers） |
| 私信发送引擎 | 生产者-消费者模型，任务队列 + 多 worker 并发消费 |
| 账号时间槽 | sync.Mutex 保护账号选取，防止并发竞争 |
| 数据存储 | 按功能拆分多个 JSON 文件，写入前加文件锁 |
| 凭证加密 | AES-256-GCM 加密，密钥基于机器码生成（绑定设备） |

### 9.3 关键依赖

| 依赖 | 用途 | 备注 |
|------|------|------|
| Wails v2 | 桌面应用框架 | Go + 前端框架 |
| 比特浏览器 API | 多账号管理 | 需用户本地安装比特浏览器 |
| 抖音 Web API | 评论/视频数据采集 | 逆向 Web 端接口，需 Cookie/Token |

### 10.4 DM 模板文件格式规范

**文件格式：** UTF-8 编码 `.txt` 文件，多条模板之间以**一个空行**分隔。

```
你好 {nickname}，看到你评论了这个视频，我这边有个不错的方案想和你分享，感兴趣的话可以聊聊～

{nickname} 你好！我是做旅行定制的，看到你对川西感兴趣，正好我有一份详细攻略，需要的话告诉我哦

Hi {nickname}，你的评论让我印象很深，我们这边有相关资源，有时间可以交流一下🙂
```

**解析规则：**

| 规则 | 说明 |
|------|------|
| 分隔符 | 连续两个 `\n`（即一个空行）作为模板边界 |
| 去空白 | 每条模板首尾的空白字符（包括换行）trim 掉 |
| 空模板过滤 | trim 后为空字符串的条目不计入模板列表 |
| 最小条数 | 至少需要 1 条有效模板，否则报错"文件中未找到有效模板" |
| 变量 | `{nickname}` → 发送时替换为目标用户昵称；未知变量保留原样 |
| 发送前检测 | 变量替换完成后执行敏感词扫描，命中则跳过（不发送，不计失败） |

**加载状态展示逻辑：**

```
文件不存在           → ❌ "模板文件不存在，请重新选择"
文件读取失败         → ❌ "文件读取失败：{错误信息}"
解析出 0 条有效模板  → ❌ "文件中未找到有效模板，请检查格式（模板间需有空行）"
解析成功 N 条        → ✅ "已加载 N 条模板"
有模板含敏感词       → ⚠️ "已加载 N 条模板（其中 M 条含敏感词，发送时将自动跳过）"
```

### 10.5 账号操作完整技术流程

#### 10.5.1 提取（Extract Cookie）完整流程

```
步骤 1：打开浏览器窗口
  → POST http://127.0.0.1:54345/browser/open {"id": profile_id}
  → 获取响应中的 ws 字段（CDP WebSocket 地址）
  → 失败则提示"比特浏览器未运行或窗口ID无效"

步骤 2：CDP 连接
  → chromium.connect_over_cdp(ws)
  → browser.contexts[0] 获取页面上下文
  → 失败则提示"CDP 连接失败，请确认比特浏览器窗口已正常打开"

步骤 3：提取 Cookie
  → context.cookies("https://www.douyin.com")
  → 过滤 domain 包含 "douyin.com" 的 Cookie 条目
  → 必须字段检查（缺任意一个则标记 expired）：
      - sessionid        ← 主登录凭证
      - sessionid_ss     ← 备份凭证
  → 可选字段提取：
      - msToken, verifyFp, fp, webid, uifid, ttwid

步骤 4：拼接 Cookie 字符串
  → 格式："key1=value1; key2=value2; ..."
  → [POC 阶段] 写入 data/cookie_store.json（见 §10.6 格式规范）
  → [生产阶段] 使用 AES-256-GCM 加密后写入 data/cookies/{profile_id}.enc

步骤 5：更新账号状态
  → 调用抖音轻量接口验证 Cookie 有效性：
      GET https://www-hj.douyin.com/aweme/v1/web/im/user/info/
      Headers: Cookie: {提取的 Cookie 字符串}
  → status_code == 0      → cookie_status = "valid"，提取账号 uid 和 nickname
  → status_code == -2     → cookie_status = "expired"，提示"Cookie 已过期，请重新登录"
  → 其他 / 超时           → cookie_status = "unknown"，提示"网络异常，请稍后重试"

步骤 6：更新签名服务
  → POST http://127.0.0.1:9527/update-cookie {"cookie": "<cookie字符串>"}
  → 成功：签名服务已使用新 Cookie 会话
  → 签名服务未运行：跳过，下次签名服务启动时自动读取 cookie_store.json

步骤 7：写回数据库（生产阶段；POC 阶段跳过此步）
  → 更新 BrowserProfile 表：cookie_status, douyin_uid, nickname, cookie_updated_at
  → 前端收到 Wails 事件 "account_updated"，刷新账号列表
```

### 10.6 Cookie 持久化文件规范（cookie_store.json）

> **适用阶段**：POC / 开发调试阶段。生产版本改用 AES-256-GCM 加密的 `data/cookies/{profile_id}.enc`。

**文件路径**：`data/cookie_store.json`

**写入方**：`tools/refresh_cookie.js`（一键刷新脚本）

**读取方**：
- `signing_server.js`（启动时加载，优先级高于 `captured_apis.json`）
- `tools/verify_all_apis.js`（API 验证工具）

**格式**：

```json
{
  "browser_id":    "9023104163bd47ffa78466d1944d0338",
  "cookie":        "sessionid=xxx; ttwid=yyy; odin_tt=zzz; ...",
  "cookie_length": 8421,
  "updated_at":    "2026-03-23T10:00:00Z"
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `browser_id` | string | 比特浏览器窗口 ID，首次输入后自动记忆 |
| `cookie` | string | 完整 Cookie 字符串，`key=value; key=value` 格式 |
| `cookie_length` | number | Cookie 字符串长度，用于快速判断完整性（< 1000 视为异常） |
| `updated_at` | ISO8601 | 最后一次刷新时间，用于提醒用户定期重新提取 |

**使用方式**（对编程小白）：

```bash
# Cookie 过期时，在项目根目录执行：
npm run refresh

# 首次运行会询问比特浏览器窗口 ID，之后自动记忆，直接执行即可
# 脚本会自动：打开窗口 → 检测登录 → 提取 → 保存 → 推送签名服务
```

## 13. 风险与合规说明

### 13.1 技术风险

| 风险 | 影响 | 缓解措施 |

### 13.2 平台合规说明

> ⚠️ **重要提示**
>
> 本工具涉及抖音平台的自动化操作，使用前请注意：
>
> 1. **平台服务条款**：抖音用户协议禁止未经授权的自动化脚本访问平台数据和接口。使用本工具可能违反平台服务条款，存在账号被封禁的风险。
> 2. **数据合规**：批量收集用户评论数据及 UID，涉及《个人信息保护法》相关条款，应在合法合规的前提下使用。
> 3. **发送频率**：过高的私信发送频率可能触发抖音的反垃圾机制，建议保持合理的发送间隔。
> 4. **用户体验**：向用户发送未经请求的私信（cold DM）可能影响用户体验，建议结合真实业务价值进行营销。

## 14. Cookie 与签名失效刷新机制

### 14.1 失效类型识别

系统运行中存在两类凭证失效场景，需分别检测和处理：

| 失效类型 | 触发特征 | 影响范围 |
|---------|---------|---------|
| **Cookie 过期** | API 返回 HTTP 302 跳转至登录页，或 `status_code = -2 / 2154` | 所有抖音 API 调用失败 |
| **msToken 过期** | API 返回 `status_code = 8` 或签名验证失败 | 评论/搜索接口失败，IM 不受影响 |
| **签名服务崩溃** | `GET /health` 连续 3 次超时或返回非 200 | a_bogus 无法生成，评论/搜索失败 |
| **签名 SDK 失效** | `/sign` 返回空 a_bogus 连续 3 次 | a_bogus 无法生成，评论/搜索失败 |

### 14.2 检测机制

#### 14.2.1 API 响应检测（Go 层）

每次调用抖音 API 后，统一检查响应：

```
响应处理流程：
1. HTTP 302 → 标记 "cookie_expired"，停止所有任务
2. status_code == -2 → 标记 "cookie_expired"
3. status_code == 8  → 标记 "mstoken_expired"（仅刷新 msToken，不强制重登）
4. 连续 3 次同一 video_id 请求失败 → 暂停该视频监控，记录错误
```

#### 14.2.2 签名服务健康检查（Go Watchdog）

独立 goroutine，每 30 秒轮询一次：

```
Watchdog 策略：
- 每 30s 调用 GET http://127.0.0.1:9527/health
- 连续 3 次失败（含超时）→ 执行重启序列：
    1. 终止旧 Node.js 进程（PID 记录在 signing_server.pid）
    2. 执行 node signing_server.js，等待最多 60s 直至 isReady=true
    3. 记录日志"签名服务已自动重启"
- 连续 3 次 a_bogus 为空 → 调用内部接口触发页面重新导航至 douyin.com
```

### 15.1 方案A：评论监控引擎

#### 15.1.1 架构设计

```
┌─────────────────────────────────────────────────────────┐
│                    评论监控引擎架构                       │
└─────────────────────────────────────────────────────────┘

  VideoQueue（SQLite）
       │
       ▼
  ┌─────────────────────────────────────────┐
  │         调度器 Scheduler                │
  │  - 按优先级排序（新视频优先）             │
  │  - 维护每个视频的下次采集时间             │
  │  - 控制全局并发数（configurable, 1-10）  │
  └──────────────────┬──────────────────────┘
                     │ 分发 VideoTask
          ┌──────────┴──────────┐
          ▼          ▼          ▼
    [Worker-1]  [Worker-2]  [Worker-N]    ← goroutine pool
          │
          ▼
    ┌─────────────────────────────┐
    │      CommentCollector       │
    │                             │
    │  1. 构建请求 URL             │
    │  2. 调用 Signer.AppendSign() │
    │  3. GET 评论列表接口          │
    │  4. 分页直到 has_more=0      │
    │  5. 对 reply_count>0 的评论   │
    │     发起回复接口（Level-2）   │
    │  6. 去重后写入 SQLite         │
    └─────────────────────────────┘
          │
          ▼
    新评论事件 → Wails EventsEmit("new_comments", data)
```

#### 15.1.2 关键设计决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 并发模型 | goroutine pool（固定大小） | 防止 goroutine 无限增长耗尽资源 |
| 分页策略 | cursor-based，有 has_more=1 则继续 | 抖音 API 原生分页机制 |
| 去重机制 | SQLite UNIQUE INDEX (comment_id) | 幂等插入，ON CONFLICT IGNORE |
| 采集深度 | Level-1 + Level-2，不超过 2 层 | 符合 PRD 需求，避免指数级 API 调用 |
| Level-2 触发条件 | `reply_comment_total > 0` | 无回复则不请求，节省 API 调用 |
| 请求限速 | 每个视频最小间隔可配（默认 30s） | 防止高频请求触发抖音限流 |
| 签名缓存 | Signer 服务内置 30s 缓存 | 相同接口复用签名，减少浏览器交互 |

#### 15.1.3 数据流

```
API 响应 comment 字段 → 内部 Comment 结构体 → SQLite comments 表

关键字段映射：
  API.cid               → comment_id
  API.text              → content
  API.create_time       → created_at
  API.user.uid          → uid
  API.user.sec_uid      → sec_uid
  API.user.nickname     → nickname
  API.ip_label          → region
  API.digg_count        → like_count
  API.reply_comment_total → reply_count
  parent_comment_id     → level-2 时填充父评论 ID，level-1 为空
```

#### 15.1.4 验证方案

| 验证阶段 | 方法 | 通过标准 |
|---------|------|---------|
| **单元测试** | Mock Signer 返回固定 a_bogus，Mock HTTP 返回预置 JSON | 正确解析评论字段，分页逻辑正确 |
| **集成测试** | 使用真实 Cookie 和 Signer，采集一个真实视频（已知评论数的视频） | DB 中评论数 ≥ API 返回 total 的 80%，不重复 |
| **Level-2 测试** | 采集一个已知有热评回复的视频 | DB 中存在 level=2 的记录，parent_comment_id 正确 |

#### 15.2.2 Protobuf Schema 逆向策略

从 `captured_im.json` 中提取的二进制 body 已知以下可读字段：

```
已确认字段（可读字符串法）：
  - 协议版本: "1.1.3"
  - app_name: "douyin_pc"
  - session_aid: "6383"
  - device_id: "8aa2dcb:Detached:8aa2dcb88b41538885168e4afbbd2b6bac8aefb2"
  - 消息内容 JSON: {"mention_users":[],"aweType":700,"richTextInfos":[],"text":"..."}
  - client_message_id: UUID 格式
  - identity_security_token: {"token":"..."}
```

逆向步骤：
1. 使用 `protoc --decode_raw` 解码已捕获的 binary payload，确定每个字段的 field number 和 wire type
2. 根据已知字符串匹配定位字段编号，构建初步 `.proto` 文件
3. 用 `google.golang.org/protobuf` 生成 Go 代码
4. 编码测试包，与捕获的原始 payload 做字节级对比验证

初步 Proto Schema（待完善）：

#### 15.2.3 identity_security_token 获取流程

```

     （请求体：公共头结构，无需特殊参数）

```

#### 15.2.4 关键设计决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 认证方式 | Cookie-only，无需 a_bogus | imapi.douyin.com 已验证无需签名 |

## 16. 比特浏览器集成方案

### 16.1 架构概述

比特浏览器（BitBrowser）是一款支持多账号隔离的指纹浏览器，每个"窗口"对应一个独立的 Chromium 进程，拥有独立的 Cookie、localStorage 和浏览器指纹。本项目利用比特浏览器管理多个已登录抖音的账号，通过其本地 API 和 CDP 协议实现：

1. **账号隔离**：每个抖音账号在独立 Chromium 进程中运行，Cookie 互不干扰
2. **Cookie 提取**：通过 CDP 直接从浏览器上下文读取抖音 Cookie
3. **签名代理**：在已登录的浏览器窗口内执行 `fetch()`，让抖音 SDK 完成 `a_bogus` 签名
4. **会话复用**：避免重复登录，利用比特浏览器的持久化 Cookie 存储

```
┌─────────────────────────────────────────────────────────┐
│                  比特浏览器集成架构                        │
└─────────────────────────────────────────────────────────┘

  用户配置的比特浏览器窗口 (BrowserProfile)
  ┌────────────┐  ┌────────────┐  ┌────────────┐
  │  窗口 A    │  │  窗口 B    │  │  窗口 C    │
  │ 抖音账号1  │  │ 抖音账号2  │  │ 抖音账号3  │
  └─────┬──────┘  └─────┬──────┘  └─────┬──────┘
        │               │               │
        └───────────────┼───────────────┘
                        │ CDP (WebSocket)
                        ▼
         ┌───────────────────────────────┐
         │     BitBrowserManager (Go)    │
         │                               │
         │  1. 调用本地 API (port 54345) │
         │  2. 获取 CDP ws 地址           │
         │  3. 通过 CDP 读取 Cookie       │
         │  4. 检测 Cookie 有效性         │
         │  5. 注入 signing_server.js    │
         └───────────────────────────────┘
                        │
                        ▼
         ┌───────────────────────────────┐
         │   签名微服务 (Node.js 9527)   │
         │   复用比特浏览器窗口 Session  │
         └───────────────────────────────┘
```

### 16.3 Cookie 有效性检测

每次打开比特浏览器窗口后，在执行任何采集/发送操作前，必须先检测抖音 Cookie 是否有效。

#### 16.3.1 检测流程

```
┌─ 打开比特浏览器窗口（/browser/open）
│
├─ 通过 CDP 连接 Playwright（connect_over_cdp）
│
├─ 读取 Cookie：从浏览器上下文获取 .douyin.com 域的所有 Cookie
│
├─ 快速校验（本地检测，无网络请求）：
│   ├─ 检查 sessionid 是否存在且非空
│   ├─ 检查 sessionid_ss 是否存在
│   └─ 如果任一缺失 → 标记为"需要重新登录"
│
└─ 深度校验（发起真实 API 请求）：
    ├─ 调用 GET /aweme/v1/web/im/user/info/（轻量接口）
    ├─ 响应 status_code == 0 → Cookie 有效
    ├─ 响应 status_code == -2 或 HTTP 302 → Cookie 过期
    └─ 网络超时 → 标记为"网络不可用"
```

#### 16.3.2 检测结果状态

| 状态 | 含义 | 处置方式 |
|------|------|---------|
| `valid` | Cookie 有效，sessionid 存在且 API 可用 | 正常使用此窗口 |
| `expired` | sessionid 缺失或 API 返回 -2 | 弹出提示，引导用户在比特浏览器中重新登录 |
| `network_error` | 接口请求超时或连接失败 | 跳过此窗口，重试其他窗口 |
| `browser_closed` | CDP 连接失败，窗口未打开 | 调用 `/browser/open` 重新打开 |

#### 16.3.3 Cookie 提取

通过 Playwright CDP 连接后，从浏览器上下文直接获取 Cookie：

```
操作步骤：
1. connect_over_cdp(ws) → 得到 browser
2. browser.contexts[0] → 得到 context
3. context.cookies("https://www.douyin.com") → 得到 cookie 数组
4. 过滤出 domain 包含 "douyin.com" 的项
5. 拼接为 "key=value; key2=value2" 格式字符串
6. 保存至数据库对应账号记录
```

### 16.5 与签名引擎集成

#### 16.5.1 Cookie 加载（当前已实现）

`signing_server.js` 启动时按以下优先级加载 Cookie：

```
优先级 1：data/cookie_store.json（npm run refresh 写入）
  → 最新的登录凭证，每次刷新自动更新
  → 缺失或无效时，回退到优先级 2

优先级 2：data/captured_apis.json（历史抓包数据）
  → 仅供初期调试，数据可能过期
  → 建议运行 npm run refresh 后不再依赖此文件
```

**热更新（运行中刷新 Cookie）：**
```
POST http://127.0.0.1:9527/update-cookie
Body: {"cookie": "<完整cookie字符串>"}
```
`tools/refresh_cookie.js` 提取完成后自动调用此接口，**无需重启签名服务**。

#### 16.5.2 签名浏览器迁移（规划中）

> 当前状态：`signing_server.js` 仍使用独立 headless Chromium 进行签名。
> 目标状态：复用比特浏览器已登录窗口，通过 CDP 连接，消除独立浏览器维护成本。

```
当前方案（已运行）：
  signing_server.js 自启 headless Chromium
  → 需要单独注入 Cookie，启动较慢

目标方案（规划中，§16 原设计）：
  signing_server.js 通过 CDP 连接比特浏览器已登录窗口
  → 复用现有 Session，Cookie 天然有效，无需单独维护
  → 实现时机：Phase 1 开发阶段
```

**目标方案修改点：**
1. 启动时调用 `/browser/open` 获取已登录窗口的 `ws`
2. 通过 `chromium.connectOverCDP(ws)` 连接
3. 后续签名逻辑不变（`page.on('request')` 拦截 + `fetch()` 触发）

**窗口选择策略：** 轮询选择 `cookie_status=valid` 的窗口，实现多账号轮换签名。

### 17.3 敏感词检测

#### 17.3.1 检测时机

在以下两个阶段执行敏感词检查：

```
阶段A（模板加载时）：
  加载 .txt 模板文件时 → 扫描所有模板内容 → 命中敏感词则高亮警告

阶段B（发送前实时检查）：
  变量替换后（如替换完 {nickname}）→ 再次扫描最终发送内容
  命中 → 跳过本条任务，记录为"敏感词拦截"状态，不算失败
```

#### 17.3.2 敏感词库管理

| 需求编号 | 需求描述 | 优先级 |
|---------|---------|--------|
| AM-020 | 内置基础敏感词库（涵盖违禁营销词、政治敏感词、抖音平台常见拦截词），随应用发布 | P0 |

| AM-023 | 检测策略：精确匹配（完整词）+ 子串匹配（词出现在句子任意位置均命中） | P0 |
| AM-024 | 检测命中时，在日志中记录："[敏感词拦截] 账号: xxx 目标: xxx 命中词: xxx" | P0 |

#### 17.3.3 检测结果处理

| 检测结果 | 处理方式 |
|---------|---------|
